# 结构性优化 · 深度评审（2026-04-12 晚）

## 上下文
经过多轮点修后，行为层次问题基本清零。这一次专门审阅**结构**：分层、依赖、命名、可测性、重复、漂移风险。给出分级重构路线，不动作。

外部观察一条：`tests/validate.js` 使用 `vm` 装载脚本并硬编码期望值（包括已被删除的 `"Claude Code"` / `"Claude Code URL Handler"` / `"Antigravity.app"` / `"Quotio.app"` 等），当前测试期望已与脚本漂移——这正是"校验/期望脱离源头"问题的又一例证，下面 #B1 的建议同时适用于测试。

---

## 深观察

### 1. 派生层把两类正交的分类压到同一 shape
`DERIVED.patterns.*` 既要承载"路由意图"（strict / media / direct），又要承载"DNS 分区"（overseas / domestic）。这两个维度是正交的：
- DIRECT 的 `overseasApps` 在 DNS 维度是 overseas；`domestic.office` 在 DNS 维度是 domestic。
- `strict.ai` / `strict.support` / `strict.validation` 都是路由意图，但 DNS 层都走 overseas——手工对齐。

`buildNameserverPolicy` 对每个派生字段手写一条 `assignNameserverPolicyDomains(..., overseas/domestic)`——**每新增一个语义类别就要新增一行**，并且必须两头改对。漂移风险中等。

### 2. 校验目标与源数据脱钩
`buildStrictValidationTargets` 里硬编码 `claude.ai / chatgpt.com / gemini.google.com / perplexity.ai / google.com / Claude / claude / codex`。源数据改了、校验忘了改——静默过关或误报。`tests/validate.js` 同病（硬编码期望规则数组）。**这是最大的漂移源**。

### 3. 规则优先级契约隐藏在 `concat` 顺序里
`buildManagedRules` 的 4 段 concat 定义了优先级：`strictChain > browserChain > media > direct`。首次出现胜出。这是**命名无法表达的运行时契约**，只有读代码才能看到。

### 4. `DERIVED` 是模块级 eager global
`buildDerivedPatterns()` 在模块加载期就跑完。消费者（`buildSnifferConfig` / `buildNameserverPolicy` / `build*Rules`）直接读 `DERIVED.patterns.*`。后果：
- 无法在不重新加载模块的情况下替换数据源做单测。
- `tests/validate.js` 不得不用 `vm` 每次全量重跑 + 硬编码期望。
- 数据与行为强耦合。

### 5. 魔法串散落在 `BASE` 之外
- `"节点选择"`（L961）硬编码在 `syncManagedGroupIntoNodeSelection`。
- `"MATCH,"`（splitMatchFallback）作字符串前缀判断。
- `"geosite:openai"`（buildNameserverPolicy）——唯一的 geosite 引用，游离。
- `BASE` 已经聚拢了 `nodeNames` / `groupNameSuffixes` / `ruleTargets` / `dns` / `errorPrefix`，继续扩到 `groupNames.nodeSelection` / `rulePrefixes.match` / `dns.openaiGeosite` 即可。

### 6. `buildDerivedPatterns` 原地改 base（L558-582）
先返回 base，再 `patterns.strict = ...; patterns.general = ...; patterns.sniffer = ...`。读者需要把函数从上到下扫一遍才能知道最终形状。建议改为返回新对象（浅合并）：base 不变，派生层纯粹。

### 7. `toSuffix` 放错段（L1446）
它在"主流程入口"段，但语义是"域名模式工具"。应挪到 `PATTERN_SHAPE` 附近。与 #3（toSuffix 锚定前缀）的修复一起考虑更合适。

### 8. `validateManagedRouting` 100+ 行过程式块（L1334-1442）
混了 5 类断言：
- target 三元关系（strictAi=chain, media≠chain, relay≠chain）
- proxy/group 存在性
- dialer 绑定
- 组内容（type + 成员集合）
- 规则命中

拆成 5 个小断言，命名即文档。

### 9. 命名动词未系统化
- `ensure*` (ensureChainGroup, ensureRegionGroup, ensureProxyContainers) — 幂等初始化/upsert
- `inject*` (injectMiyaProxies, injectManagedRules) — 写入
- `apply*` (applyDnsAndSniffer, applyManagedRouting) — 装配步骤
- `resolve*` (resolveRelayTarget, resolveRegionMeta, resolveRoutingTargets) — 读取并计算
- `build*` — 纯函数生产

`ensureChainGroup` 实际等于 `upsert chain group`，与 `inject*` 语义重叠。可统一为：纯 `build*` + 写入 `writeInto*(config, ...)` 两类。

### 10. DNS 字段冗余遍历
`buildRuleIdentityLookup(managedRules)` + `dedupeRulesByIdentity(concatenated)` + `filterConflictingRules(...)` + `buildStringLookup(config.rules)`（validate 阶段）——每阶段都重扫一遍规则。每条规则只有几百条，总耗时 <5ms，**无需优化性能**，但值得列出，看是否可合并一次 pass。

### 11. 工具函数层面健康
`uniqueStrings` / `mergeStringGroups` / `excludeStrings` / `flattenGroupedPatterns` / `buildStringLookup` / `haveSameStringSet` 命名清晰、单职责、总计 <50 行。不是 sprawl，保留。
